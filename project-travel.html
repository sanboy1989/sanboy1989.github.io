Context & Problem

Why traditional travel agencies are losing trust

傳統旅行社為何正在失去客戶信任

A travel agent manages dozens of trips simultaneously. Every client has a different flight, different hotel, different itinerary. When a flight time changes at 2am, the agent needs to notify 8 different clients — manually, one by one, via WhatsApp. The chain breaks constantly: messages get missed, calendars don't get updated, clients show up at the wrong terminal.

旅行社代理同時管理數十個行程。每位客戶有不同的航班、不同的酒店、不同的行程。當航班時間在凌晨 2 點更改時，代理需要逐一手動通知 8 位不同客戶——通過 WhatsApp。這條鏈條不斷斷裂：消息被遺漏、日曆未更新、客戶走錯航廈。

WanderSync replaces that manual chain with automated workflows. When an agent books or modifies any itinerary item, the system propagates the change — updating the client's Google Calendar, sending a formatted email summary, and firing a push notification — all within seconds, without touching a phone.

WanderSync 用自動化流程取代了那條手動鏈條。當代理預訂或修改任何行程項目時，系統自動傳播變更——更新客戶的 Google 日曆、發送格式化郵件摘要、觸發推送通知——全部在數秒內完成，無需觸碰手機。

Tech Deep Dive

The Stack & the Hard Parts

技術棧與解決難題

Next.js with SSR is the right choice here because the client portal needs SEO for booking pages, and server-side rendering keeps itinerary data out of the client bundle. Node.js handles the Google Calendar API calls and notification dispatch on the backend. Nodemailer sends structured itinerary emails; the notification layer uses a webhook pattern rather than polling.

Next.js 配合 SSR 是正確選擇，因為客戶端入口需要為預訂頁面提供 SEO，而伺服器端渲染使行程數據遠離客戶端捆綁包。Node.js 在後端處理 Google Calendar API 調用和通知分發。Nodemailer 發送結構化行程郵件；通知層使用 Webhook 模式而非輪詢。

Key Design Decision
關鍵設計決策
Admin CRM panel as the single source of truth
Admin CRM 面板作為唯一真實來源

All itinerary changes flow through the admin CRM panel, never directly to the client. This creates an audit trail — every change is logged with a timestamp and agent ID — and prevents partial updates where the client's calendar is ahead of or behind the agent's records.

所有行程變更都通過 Admin CRM 面板流轉，而非直接到達客戶。這創建了一條審計追蹤——每次變更都記錄了時間戳和代理 ID——並防止客戶日曆超前或落後於代理記錄的部分更新。

Hardest Bug
最難解決的 Bug

Duplicate calendar events under high-concurrency conditions. When an agent updated an itinerary and the client simultaneously opened the portal, two Google Calendar API writes fired in parallel — one from the agent save, one from the portal's onMount sync. The Calendar API created two separate events instead of updating the existing one. Fixed by introducing idempotency keys: every itinerary item has a stable UUID used as the Calendar event ID, so any write is always an upsert rather than a blind create.

高併發條件下出現重複日曆事件。當代理更新行程且客戶同時打開入口網站時，兩個 Google Calendar API 寫入並行觸發——一個來自代理保存，另一個來自入口網站的 onMount 同步。Calendar API 創建了兩個獨立事件，而非更新現有事件。解決方案是引入冪等鍵：每個行程項目有一個穩定的 UUID 用作日曆事件 ID，因此任何寫入都是更新插入（upsert）而非盲目創建。
